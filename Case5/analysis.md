# Анализ качества проекта Case4: веб-калькулятор

## Введение

Case4 представляет собой одностраничное веб-приложение для выполнения четырёх базовых арифметических операций: сложения, вычитания, умножения и деления. Пользователь вводит два значения в текстовые поля и выбирает операцию кнопкой. Логика размещена в `script.js`, внешний вид — в `style.css`, а разметка и элементы управления — в `index.html`.

Несмотря на небольшой размер, проект хорошо подходит для анализа качества программного продукта. В нём есть ввод данных, преобразование строк в числа, обработка ошибок, пользовательская обратная связь и отделение представления от логики. Ниже оцениваются сильные стороны текущего решения, его ограничения и практические пути развития.

## 1. Функциональность

Базовая функциональность реализована полностью. На странице есть два поля ввода и четыре подписанные кнопки: «Сумма», «Разность», «Произведение» и «Деление». После нажатия кнопки программа считывает оба значения, преобразует запятую в точку, проверяет, что значения являются конечными числами, и выполняет выбранную операцию. Результат выводится в отдельный элемент `output`.

Особенно важна обработка нештатных ситуаций. Пустое поле, текст вместо числа или значение, которое JavaScript не может преобразовать в конечное число, приводят к сообщению «Ошибка! Введите числа.». При делении на ноль показывается отдельное понятное сообщение «Деление на ноль невозможно.». Благодаря этому приложение не выдаёт пользователю технические значения `NaN` или `Infinity`.

Текущая реализация поддерживает как точку, так и запятую в качестве десятичного разделителя. Это удобно для русскоязычной аудитории. Все кнопки подключены через `addEventListener`, поэтому обработчики не смешиваются с HTML-разметкой и могут быть легко расширены.

Ограничение функциональности состоит в том, что калькулятор рассчитан только на две операции над двумя числами. Он не поддерживает цепочки выражений, проценты, отрицательные степени, историю результатов, клавиатурный ввод и форматирование очень длинных чисел. Для учебной задачи это приемлемо: набор возможностей точно соответствует заявленным требованиям. Для реального пользовательского инструмента желательно добавить историю вычислений, кнопку очистки и управление с клавиатуры.

## 2. Производительность

Производительность приложения очень высокая для его масштаба. В документе всего несколько DOM-элементов, а вычисления имеют постоянную сложность O(1). Нажатие любой кнопки выполняет разбор двух строк, несколько условных проверок и обновление одного текстового узла. Даже на слабом мобильном устройстве такая операция выполняется практически мгновенно.

Стили также не создают заметной нагрузки. Градиенты, тени и небольшие переходы применяются к ограниченному числу элементов. События назначаются один раз при загрузке страницы. Нет сетевых запросов, больших изображений, внешних библиотек или циклов, которые могли бы блокировать интерфейс.

Возможная зона внимания — анимации теней и трансформаций на очень старых устройствах. Однако они используются только при наведении и не затрагивают вычисления. Если проект будет включён в тяжёлую страницу с большим количеством компонентов, полезно будет предпочесть анимацию `transform` и `opacity`, как в текущем решении, а не анимацию размеров и координат.

Для дальнейшей оптимизации здесь не требуется преждевременное усложнение. Гораздо полезнее сохранить отсутствие зависимостей и малый размер файлов. При добавлении истории вычислений следует ограничить количество отображаемых записей или использовать постраничный вывод, чтобы бесконтрольный рост DOM не ухудшал отзывчивость.

## 3. Юзабилити

Интерфейс построен вокруг одного понятного сценария: ввести два числа и выбрать действие. Подписи полей находятся над полями ввода, а не заменены только плейсхолдерами. Это повышает понятность формы и помогает пользователю, когда поле уже заполнено. Кнопки расположены сеткой два на два; все операции видны одновременно и не требуют дополнительной навигации.

Визуальная иерархия удачна: заголовок объясняет назначение страницы, поля имеют заметные состояния фокуса, а блок результата отделён контрастным светлым фоном. Успешный результат, ошибки ввода и ошибка деления на ноль окрашиваются по-разному. Пользователь получает обратную связь в том же месте, где ожидает увидеть ответ.

Атрибут `aria-live="polite"` у результата делает динамические сообщения доступными для пользователей программ чтения с экрана. Тип ввода `text` выбран намеренно: он позволяет корректно принять и точку, и запятую. Атрибут `inputmode="decimal"` при этом помогает мобильному браузеру показать числовую клавиатуру.

Есть несколько способов улучшить удобство. Следует вычислять результат при нажатии клавиши Enter в любом поле. Полезны кнопка «Очистить» и возможность быстро скопировать ответ. Можно добавлять понятный формат чисел: например, не показывать лишние нули после запятой и отделять большие разряды пробелами. Также стоит сообщать об ошибке сразу после потери фокуса в поле, а не только после выбора операции. При этом не нужно блокировать ввод промежуточных значений, например одиночного минуса, пока пользователь ещё печатает.

## 4. Безопасность

Проект не хранит данные, не имеет сервера и не выполняет внешних запросов. Поэтому поверхность атаки минимальна. Данные считываются через свойство `value`, а результат выводится через `textContent`, а не через `innerHTML`. Это важная деталь: если пользователь введёт строку с HTML-тегами или сценарием, она не будет интерпретирована браузером как код.

Преобразование выполняется стандартной функцией `Number`, а проверка `Number.isFinite` отсеивает `NaN`, бесконечность и нечисловые значения. Благодаря этому исключаются некорректные арифметические результаты. Проверка делителя на равенство нулю предотвращает выдачу бесконечности.

Ограничения безопасности связаны главным образом с будущим развитием. Если появится сохранение истории в `localStorage`, данные должны рассматриваться как недоверенные при повторном выводе. Если будет добавлен серверный API, проверка чисел должна дублироваться на сервере, а страницы должны обслуживаться по HTTPS. При добавлении внешних скриптов следует применять Content Security Policy и фиксировать версии зависимостей. Для текущего автономного приложения этих мер не требуется, но архитектурно важно не заменять `textContent` на небезопасный `innerHTML` без экранирования данных.

## 5. Масштабируемость

Код организован достаточно понятно для текущего объёма. Константы и ссылки на DOM-элементы находятся в начале файла, парсинг вынесен в `parseNumber`, отображение — в `showResult`, а выполнение операции — в `calculate`. Такая декомпозиция позволяет добавить новые операции без полного переписывания страницы.

Тем не менее при росте функций длинная цепочка `if` в `calculate` станет неудобной. Для расширяемой версии лучше использовать объект операций. В нём строковый идентификатор кнопки будет сопоставлен с функцией вычисления и человекочитаемым названием. Тогда добавление возведения в степень, остатка от деления или округления потребует изменения в одном месте. Для операций с одним аргументом, таких как квадратный корень, полезно выделить отдельный режим или более универсальную модель выражений.

Отдельный следующий шаг — отделить доменную логику от DOM. Функции сложения, вычитания, умножения, деления и валидации могут находиться в отдельном модуле без обращения к странице. Это сделает их пригодными для автоматических тестов и повторного использования в мобильном приложении, React-компоненте или серверной части.

Для учебного калькулятора нет необходимости подключать фреймворк. Использование React, Vue или другой библиотеки увеличит сложность сборки без заметной пользы. Масштабирование должно быть постепенным: сначала выделение модулей и тесты, затем история операций и хранение состояния, и только после появления сложного интерфейса — решение о фреймворке.

## 6. Сопровождаемость

Сопровождаемость обеспечивается простой файловой структурой. HTML отвечает за семантическую разметку, CSS — за оформление, JavaScript — за поведение. Имена переменных понятны: `firstNumberInput`, `secondNumberInput`, `operationButtons`, `calculationResult`. Константы операций передаются через `data-operation`, поэтому связь кнопок с логикой легко найти.

Функция `showResult` централизует обновление сообщения и CSS-класса. Это предотвращает повторение кода и снижает риск, что один из сценариев забудет изменить оформление. Функция `parseNumber` собирает правила обработки десятичного разделителя в одном месте. Если правила локали изменятся, их можно скорректировать без поиска по всему проекту.

Для улучшения сопровождения стоит добавить краткие комментарии только к нетривиальным решениям: например, объяснить, почему `input` имеет тип `text` и почему используется `Number.isFinite`. Полезно также настроить форматирование и проверку кода с помощью ESLint и Prettier. Они помогут обнаруживать случайные неиспользуемые переменные и сохранять единый стиль при работе нескольких разработчиков.

README можно расширить разделами «Поддерживаемые форматы чисел», «Сценарии ошибок» и «План развития». Это сократит время знакомства нового участника с проектом. При появлении модулей стоит описать публичные функции и их ожидаемые аргументы через JSDoc.

## 7. Переносимость

Калькулятор использует стандартные возможности современных браузеров: HTML5, CSS Grid, `addEventListener`, `const`, стрелочные функции, шаблонные строки и `Number.isFinite`. Он запускается без установки Node.js, сервера или пакетов: достаточно открыть `index.html`. Такой подход хорошо переносится между Windows, macOS, Linux, Android и iOS.

Адаптивная часть интерфейса учитывает узкие экраны. При ширине до 420 пикселей сетка кнопок меняется с двух колонок на одну, а внутренние отступы уменьшаются. Это предотвращает горизонтальную прокрутку и сохраняет размер сенсорных целей комфортным для касания.

Главное ограничение — очень старые браузеры, не поддерживающие CSS Grid или современный синтаксис JavaScript. Для современной учебной работы это допустимо. Если потребуется поддержка старых корпоративных окружений, можно заменить Grid на Flexbox, транспилировать JavaScript и использовать полифиллы. Однако это добавит инструменты сборки и не оправдано без конкретного требования.

## 8. Качество кода

Код соответствует базовым принципам читаемости. Каждая функция имеет одну чёткую ответственность: разбор значения, показ результата или вычисление. Повторяющиеся действия сведены к минимуму. Нет глобальных обработчиков в HTML и нет внешних библиотек, которые осложняют аудит или запуск.

Валидация построена корректно: `Number.isFinite` лучше, чем глобальная `isNaN`, потому что не выполняет неочевидного преобразования аргументов. Проверка деления на ноль находится до операции, поэтому пользователь получает предметное сообщение вместо технического результата. Функция `parseNumber` сначала удаляет пробелы, затем нормализует десятичный разделитель.

Есть возможность сделать код ещё надёжнее. В текущем виде `replace(",", ".")` заменяет только первую запятую. Для одного числа этого достаточно, но строка с несколькими запятыми будет отклонена `Number`, что является безопасным поведением. Более строгий вариант может использовать регулярное выражение и явно проверять допустимый формат числа. Также полезно отформатировать результат через `Intl.NumberFormat("ru-RU")`, чтобы избежать неаккуратного вывода дробей с особенностями двоичной арифметики, например `0.30000000000000004`.

Кнопки содержат русские текстовые подписи, а для программы чтения с экрана доступны названия элементов. Это хороший баланс между визуальным и семантическим качеством. Дальнейший шаг — добавить `aria-describedby` для связи полей с блоком подсказки или ошибки и проверить достаточность контраста цветов по стандарту WCAG.

## 9. Тестирование

В текущей статической версии автоматические тесты не приложены, поэтому проверка выполняется вручную в браузере. Для минимального набора ручных проверок необходимо открыть страницу и убедиться, что каждая операция работает с целыми и дробными числами, что запятая принимается как разделитель, что нечисловой текст вызывает ошибку, а деление на ноль имеет отдельное сообщение. Важно проверить интерфейс на узком экране и перейти по элементам клавишей Tab.

Для систематической проверки стоит выделить вычислительные функции в отдельный модуль и покрыть их тестами. Например, можно проверить, что 2 + 3 равно 5, 7 − 10 равно −3, 2.5 × 4 равно 10, а деление на ноль возвращает объект ошибки. В браузерных тестах следует проверить, что нажатие каждой кнопки вызывает ожидаемое обновление `output`, а неправильный ввод добавляет класс `error`.

Подходящими инструментами будут Vitest или Jest для модульных тестов и Playwright для сценариев в браузере. Для небольшого проекта также полезна проверка без библиотек: вынести чистые функции в модуль и выполнять тестовый файл через Node.js. При добавлении тестов нужно фиксировать не только обычные примеры, но и крайние случаи: отрицательные числа, ноль, пробелы вокруг числа, дроби с запятой, очень большие значения и строку из пробелов.

## Рекомендации по улучшению

1. Вынести арифметические операции и проверку входных данных в отдельный JavaScript-модуль, чтобы покрыть их модульными тестами.
2. Добавить поддержку клавиши Enter, кнопку очистки полей и фокус на первом поле после очистки.
3. Использовать объект операций вместо цепочки условий, чтобы легко добавлять новые действия.
4. Форматировать вывод с учётом русской локали и разумного ограничения числа знаков после запятой.
5. Реализовать историю последних вычислений с возможностью повторного использования результата.
6. Добавить ESLint, Prettier и автоматические тесты в GitHub Actions для контроля качества при каждом изменении.
7. Проверить контраст, навигацию с клавиатуры и сообщения скринридера по требованиям доступности WCAG.
8. При появлении серверной части дублировать проверку данных на сервере, применять HTTPS и не выводить пользовательский ввод через `innerHTML`.

## Заключение

Case4 — компактный, полностью работоспособный калькулятор, который хорошо решает поставленную учебную задачу. Его сильные стороны — прозрачная структура, понятная обработка ошибок, современный адаптивный интерфейс и отсутствие лишних зависимостей. Небольшой размер не мешает проекту демонстрировать важные инженерные практики: валидацию данных, разделение ответственности и доступную обратную связь.

Наиболее ценные улучшения для следующего этапа — тестирование чистой вычислительной логики, улучшение клавиатурного сценария и форматирование чисел. Эти изменения заметно повысят надёжность и удобство, сохранив простоту проекта.
